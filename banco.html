🎛️ ENUM vs CHECK

Subquery & CTE
Aula 3 — consultas aninhadas e expressões de tabela comuns
🧩 O que é uma Subquery?

Uma subquery é uma consulta SQL aninhada dentro de outra. Sempre dentro de parênteses (). Pode retornar um único valor, uma linha ou várias linhas. Pode aparecer em SELECT, FROM ou WHERE.

📍 Subquery no WHERE (mais comum)
SELECT nome, salario FROM empregados WHERE salario > (SELECT AVG(salario) FROM empregados);

A subquery calcula a média primeiro, depois filtra quem ganha acima dela.

📊 Subquery no SELECT
SELECT nome, salario, (SELECT AVG(salario) FROM empregados) AS media_geral FROM empregados;

Cada linha exibe o salário do funcionário e a média geral do lado.

🗂️ Subquery no FROM (tabela temporária)
SELECT * FROM ( SELECT departamento, AVG(salario) AS media FROM empregados GROUP BY departamento ) AS subquery;

Cria uma "tabela temporária virtual" que a query externa consulta.

✅ Quando USAR subquery

• Calcular valores intermediários
• Quando retorna um único valor
• Filtragem condicional complexa
• Quando é mais legível que JOIN

❌ Quando NÃO usar

• Tabelas grandes (pode ser lento)
• Quando um JOIN resolve
• Subqueries aninhadas profundas
• Dificuldade de indexação

🏷️ CTE — Common Table Expression

Uma CTE cria uma consulta temporária nomeada usando WITH. Mais legível que subquery, pode ser reutilizada na mesma query.

WITH funcionarios_bem_pagos AS ( SELECT nome, salario FROM funcionarios WHERE salario > 5000 ) SELECT * FROM funcionarios_bem_pagos;
🔄 CTE Recursiva (dados hierárquicos)
WITH RECURSIVE hierarquia AS ( -- Âncora: o chefe principal (sem chefe) SELECT id, nome, chefe_id, 1 AS nivel FROM funcionarios WHERE chefe_id IS NULL UNION ALL -- Recursão: busca os subordinados SELECT f.id, f.nome, f.chefe_id, h.nivel + 1 FROM funcionarios f JOIN hierarquia h ON f.chefe_id = h.id ) SELECT * FROM hierarquia ORDER BY nivel;

Útil para organogramas e categorias hierárquicas.

✅ Quando USAR CTE

• Consultas complexas com legibilidade
• Subqueries repetitivas
• Cálculos intermediários
• Dados hierárquicos (WITH RECURSIVE)

❌ Quando NÃO usar CTE

• Se um JOIN simples resolve
• No MySQL, CTEs não são materializadas (pode ser mais lento que temp table)
• Conjuntos de dados muito grandes

💡 Diferença chave: Subquery no FROM precisa de alias (AS subquery). CTE nomeia a consulta antes com WITH nome AS (...).

CASE WHEN
Aula 4 — condicional SQL equivalente ao if/else
📐 Estrutura básica
CASE WHEN condicao1 THEN resultado1 WHEN condicao2 THEN resultado2 ELSE resultado_padrao END AS nome_coluna
🎯 CASE Pesquisado

Cada WHEN tem uma condição completa com operadores:

CASE WHEN salario >= 5000 THEN 'Alto' WHEN salario >= 3000 THEN 'Médio' ELSE 'Baixo' END
🎯 CASE Simples

Compara uma expressão com valores fixos:

CASE status WHEN 'A' THEN 'Ativo' WHEN 'I' THEN 'Inativo' ELSE 'Desconhecido' END
📋 Regras importantes

• O SQL avalia as condições de cima para baixo e retorna o primeiro WHEN verdadeiro
• Se nenhuma condição for satisfeita → retorna ELSE
• Se não houver ELSE → resultado é NULL
• O END é obrigatório

🗂️ CASE no GROUP BY
SELECT CASE WHEN salario >= 5000 THEN 'ALTO' WHEN salario >= 3000 THEN 'MÉDIO' ELSE 'BAIXO' END AS faixa_salarial, COUNT(*) AS total_funcionarios FROM funcionarios GROUP BY faixa_salarial;
📊 CASE no ORDER BY (ordenação personalizada)
ORDER BY CASE WHEN departamento = 'TI' THEN 1 WHEN departamento = 'RH' THEN 2 ELSE 3 END

Ordena por prioridade de negócio, não alfabeticamente.

Window Functions
Aula 4 — funções analíticas (MySQL 8.0+)
🎯 O conceito fundamental

GROUP BY → agrupa e reduz linhas
Window Function → calcula sobre um grupo mantendo todas as linhas

💡 Analogia: Window function é como calcular a média do departamento e mostrar ela do lado de cada funcionário individualmente.
⚙️ Estrutura
funcao() OVER ( PARTITION BY coluna -- divide em grupos ORDER BY coluna -- ordena dentro do grupo )

OVER() vazio = a janela é a tabela toda.

📊 Funções de Agregação com OVER
FunçãoO que fazExemplo de uso
SUM() OVERTotal acumulado por partiçãoVendas acumuladas por vendedor
AVG() OVERMédia do grupo na mesma linhaMédia salarial do departamento ao lado de cada nome
COUNT() OVERContagem do grupo repetidaQtde de funcionários do depto em cada linha
🏆 Funções de Ranking — a diferença crucial
FunçãoEmpate?Resultado com empate
ROW_NUMBER()Ignorado1, 2, 3, 4 — sempre único
RANK()Posição repetida + salto1, 2, 2, 4 (pula o 3)
DENSE_RANK()Posição repetida sem salto1, 2, 2, 3 (sem pular)
⚠️ Dica de prova: RANK pula posição após empate. DENSE_RANK não pula. ROW_NUMBER nunca empata.
⬅️ LAG() — valor anterior
LAG(valor) OVER ( PARTITION BY vendedor_id ORDER BY data_venda ) AS valor_anterior

Compara a linha atual com a linha anterior. Primeira linha → NULL.

➡️ LEAD() — próximo valor
LEAD(valor) OVER ( PARTITION BY vendedor_id ORDER BY data_venda ) AS proxima_venda

Compara a linha atual com a linha seguinte. Última linha → NULL.

Índices & EXPLAIN
Aula 5 — performance e análise de execução
📚 O que é um índice?

Índice é uma estrutura de dados separada (B-Tree) que mapeia valores de coluna para a localização física das linhas — como o índice remissivo de um livro. Evita varredura completa da tabela (Full Table Scan).

WHERE JOIN ORDER BY GROUP BY BETWEEN / > / <
📋 Tipos de Índice no MySQL
TipoSintaxeCaracterística
PRIMARY KEYautomáticoJá vem com índice, sempre único
UNIQUE INDEXCREATE UNIQUE INDEXAcelera busca + proíbe duplicatas
INDEX simplesCREATE INDEXB-Tree padrão em uma coluna
Índice CompostoCREATE INDEX ON (a,b)Duas colunas — obedecer ordem!
📏 As 4 Regras para criar índices
Regra A — Alta Cardinalidade (Seletividade)

Crie em colunas com muitos valores únicos. Bom: cpf, email. Ruim: sexo, status — MySQL ignora o índice se retornar >30% das linhas.

Regra B — Prefixo à Esquerda (Leftmost Prefixing)

Para índices compostos (A, B, C): buscas por A, A+B, A+B+C usam o índice. Busca só por B ou C não usa.

Regra C — Funções Matam Índice ("Mata-Índice")

❌ Errado: WHERE YEAR(data_cadastro) = 2023
✅ Certo: WHERE data_cadastro >= '2023-01-01' AND data_cadastro <= '2023-12-31'

Regra D — Índice melhora leitura, custa escrita

Cada INSERT, UPDATE, DELETE também atualiza o índice. Mais índices = mais custo de manutenção. Não indexe tudo.

🔬 EXPLAIN — entendendo o plano de execução

Coloque EXPLAIN antes de um SELECT e o MySQL mostrará o plano — sem executar a query.

Coluna type — a mais importante, do melhor ao pior:
typeSignificadoNível
const / eq_refUsou PK ou Unique → vai direto na linha🟢 Perfeito
refUsou índice B-Tree normal → poucas linhas🟢 Ótimo
rangeIntervalo com índice (>, <, BETWEEN)🟡 Bom
indexLeu a árvore B-Tree inteira (Index Scan)🟠 Atenção
ALLFull Table Scan — ignorou índices🔴 Péssimo
⚠️ Pegadinha: Em tabelas com poucas linhas o MySQL pode retornar ALL mesmo com índice criado — ele calcula que é mais rápido ler tudo. Isso é normal.
🗺️ Outras colunas do EXPLAIN

possible_keys — índices que poderiam ajudar
key — índice que o MySQL escolheu usar
rows — estimativa de linhas que serão lidas
filtered — % de linhas que passam pelo WHERE
Extra — avisos: Using filesort, Using temporary, Using index

Views
Aula 5 — tabelas virtuais que simplificam e protegem
🎯 O que é uma View?

Uma View é uma tabela virtual que armazena apenas o código SQL (não os dados). Cada consulta sobre ela executa a query original por baixo dos panos.

CREATE VIEW vw_nome AS SELECT c.nome, p.data_pedido, prod.nome AS produto FROM clientes c INNER JOIN pedidos p ON c.id = p.cliente_id INNER JOIN produtos prod ON p.produto_id = prod.id; -- Consultar é simples: SELECT * FROM vw_nome;
🏆 Os 3 superpoderes de uma View
1. Simplicidade — esconder complexidade

Aquela query gigante com 10 JOINs vira um simples SELECT * FROM vw_relatorio_vendas.

2. Segurança — restringir acesso a colunas

Telemarketing vê nome/email/telefone, mas não vê senha. Logística não vê valor financeiro. Você dá permissão só para a View, não para a tabela original.

3. Consistência e manutenção

Regra de negócio muda? Altera uma vez na View. O sistema inteiro reflete a mudança.

⚠️ Limitações importantes

• Views não melhoram performance — são só SQL salvo. Os índices precisam estar nas tabelas base.

• Views atualizáveis (simples, sem JOIN): você pode fazer INSERT/UPDATE/DELETE através da View.

• Views somente leitura (com JOIN, GROUP BY, HAVING, agregações, DISTINCT, UNION): não pode alterar dados.

🔧 Comandos úteis para ver Views e Índices
-- Listar índices de uma tabela SHOW INDEX FROM pedidos; -- Listar todas as views do banco SHOW FULL TABLES IN banco WHERE TABLE_TYPE = 'VIEW';

Q1Para armazenar valores monetários como salários, qual tipo é o mais adequado?
DECIMAL armazena números com precisão fixa, evitando erros de arredondamento. FLOAT e DOUBLE usam precisão aproximada — para dinheiro isso é inaceitável.
Q2Uma subquery no WHERE retorna a média salarial. Qual funcionário vai aparecer no resultado abaixo?
WHERE salario > (SELECT AVG(salario) FROM empregados)
Salários: Ana=4000, Bruno=5500, Carlos=3000, Daniela=3500, Eduardo=7000, Fernanda=6500. Média ≈ 4916
Média ≈ 4916,66. Somente Bruno (5500), Eduardo (7000) e Fernanda (6500) estão acima. Ana (4000) está abaixo da média.
Q3No CASE WHEN, o que acontece se nenhuma condição for verdadeira e não houver cláusula ELSE?
Quando nenhuma condição é satisfeita e não há ELSE, o resultado é NULL. Por isso é boa prática sempre incluir um ELSE como fallback.
Q4Qual a diferença entre RANK() e DENSE_RANK() quando dois funcionários empatam na posição 3?
RANK() pula a posição após empate (1,2,3,3,5). DENSE_RANK() não pula (1,2,3,3,4). ROW_NUMBER() nunca empata, sempre único.
Q5Qual query usa corretamente um índice criado em data_cadastro?
Colocar a coluna dentro de uma função (YEAR, MONTH, DATE_FORMAT) "mata o índice" — o MySQL não consegue usar a estrutura B-Tree. A forma correta usa intervalos diretos na coluna.
Q6Você criou um índice composto (cidade, estado). Qual busca NÃO vai usar o índice?
Regra do Prefixo à Esquerda: o índice (cidade, estado) só é usado se a query incluir a coluna mais à esquerda — neste caso, 'cidade'. Buscar apenas por 'estado' ignora o índice.
Q7O que significa type: ALL no resultado de um EXPLAIN?
type: ALL é o pior cenário — significa que o MySQL leu cada linha da tabela do início ao fim. Em tabelas grandes isso derruba o servidor. O objetivo é ter type: const, eq_ref ou ref.
Q8Uma View com GROUP BY e SUM() é considerada:
Views com GROUP BY, HAVING, funções de agregação (SUM, COUNT), DISTINCT ou UNION são somente leitura (Read-Only). Apenas Views simples, baseadas em uma tabela sem essas operações, permitem INSERT/UPDATE/DELETE.
Q9Qual a principal vantagem da CTE sobre a Subquery no FROM?
A CTE nomeia a consulta, tornando o código mais legível e permitindo referenciar o mesmo resultado em múltiplos lugares da query. No MySQL a CTE não é materializada (performance similar à subquery).
Q10A função LAG() em uma Window Function serve para:
LAG() retorna o valor da linha anterior dentro da partição (definida por PARTITION BY). É útil para comparar uma venda com a venda anterior do mesmo vendedor. A primeira linha retorna NULL.
Q11Qual tipo de dado ocupa 1 byte e armazena apenas o ano (YYYY)?
YEAR (específico do MySQL) ocupa 1 byte e armazena apenas o ano no formato YYYY. DATE ocupa 3 bytes e guarda data completa. TINYINT ocupa 1 byte mas é numérico inteiro.
Q12Para criar uma CTE que percorra uma hierarquia de chefes e subordinados, qual palavra-chave adicional é necessária?
WITH RECURSIVE permite que a CTE referencie a si mesma. A estrutura precisa de uma âncora (registro raiz, ex: chefe sem chefe_id) + uma parte recursiva que faz JOIN com a CTE até não encontrar mais subordinados.
